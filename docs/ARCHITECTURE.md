# Micro Mind Architecture

## 1. Purpose

Micro Mind is a Python-first evolutionary task execution engine.

It is not an LLM replacement and it is not a coding assistant by itself.

Micro Mind sits above tools, local models, cloud models, command executors and future agents.

Its purpose is to:

- receive a user goal
- classify the goal
- activate the correct workflow network
- execute deterministic steps
- verify results
- save memory
- improve future executions over time

The first version must stay small and deterministic.

---

## 2. Core Separation

Micro Mind has two separate worlds.

### 2.1 Engine

The engine is this repository.

It contains:

- planners
- networks
- nodes
- memory readers/writers
- verification logic
- future evolution logic

The engine must not be confused with the projects it creates.

### 2.2 Output Projects

Output projects are created outside the engine.

Example:

```text
/Volumes/HDD/MM_projects/block_heaven
```

These folders are generated by Micro Mind workflows.

They may contain:

- frontend
- backend
- database
- docs
- PROJECT_STATE.md
- DEVELOPMENT_RULES.md
- .micro_mind/

The `.micro_mind/` folder inside each output project stores project-specific execution memory.

Phase 1.3 - ExecutionContext

- create project structures

- write execution memory
- persist node statistics

## 4. Implemented Networks

### 4.1 ProjectCreateNetwork

Node chain:

```text
RootPlanner
→ ProjectCreateNetwork
→ QuestionNode
→ DirectoryCreateNode
→ StructureVerifyNode
→ MemoryNode
```

Purpose:

Create a deterministic project structure and store execution memory.

CLI:

```bash
python3 main.py create-project \
  --name "Block Heaven" \
  --type flutter \
  --target /Volumes/HDD/MM_projects
```

Expected output project:

```text
/Volumes/HDD/MM_projects/block_heaven/
├── frontend/
├── backend/
├── database/
├── docs/
├── PROJECT_STATE.md
├── DEVELOPMENT_RULES.md
└── .micro_mind/
    ├── executions/
    └── nodes/
```

### 4.2 MemoryReaderNetwork

Node chain:

```text
RootPlanner
→ MemoryReaderNetwork
→ MemoryReadNode
→ MemorySummaryNode
```

Purpose:

Read existing project memory and produce a summarized view of project history.

CLI:

```bash
python3 main.py memory-summary \
  --project /Volumes/HDD/MM_projects/block_heaven
```

Expected summary fields:

```text
project_root
total_executions
successful_executions
failed_executions
last_execution
tasks_seen
node_stats
most_used_nodes
average_duration_seconds
```

## 5. Node Model

Every node must have lifecycle statistics.

Required fields:

```text
node_id
node_name
activation_count
success_count
failure_count
last_execution
sleep_score
fitness_score
state
```

Allowed states:

```text
ACTIVE
SLEEPING
PROMOTED
DEPRECATED
ARCHIVED
```

Current behavior:

- Nodes activate during execution.
- Successful nodes increase success count.
- Failed nodes increase failure count.
- Finished nodes return to sleeping state.
- Fitness score is calculated from success/failure ratio.

---

## 6. Network Model

A network is an ordered workflow made of nodes.

A network must:

- know its node chain
- pass context between nodes
- collect outputs
- stop safely on failure
- return a structured result

Implemented networks:

```text
ProjectCreateNetwork
MemoryReaderNetwork
```

ProjectCreateNetwork now uses a shared `ExecutionContext` internally so nodes
can update one workflow state object instead of passing loose dictionaries at
each step. The CLI response remains compatible with earlier Phase 1 output.

Future networks:

```text
FlutterCreateNetwork
GitSetupNetwork
FirebaseSetupNetwork
ReleaseNetwork
DeployNetwork
```

## 7. Memory Model

Each output project has its own memory directory:

```text
<project_root>/.micro_mind/
├── executions/
└── nodes/
```

### 7.1 Executions

Execution records are stored as JSON.

They describe what happened during a workflow run.

Required fields:

```text
schema_version
task
project_name
project_type
target_directory
project_root
result
nodes_used
duration_seconds
created_directories
created_files
verification_result
timestamp
```

### 7.2 Node Statistics

Node statistics are stored as JSON files.

Example:

```text
.micro_mind/nodes/DirectoryCreateNode.json
```

These records allow Micro Mind to understand which nodes are reliable over time.

---

## 8. Execution Principle

Micro Mind should grow in small deterministic steps.

Each phase must follow this rule:

```text
Plan → Execute → Verify → Remember
```

No feature is complete until it can:

- run from CLI
- verify its own result
- write/read memory where appropriate
- be tested
- be repeated safely

Purpose:

- inspect last execution

---

## 10. What Micro Mind Is Not Yet

Micro Mind currently cannot:

- write application code
- run Flutter create
- initialize Git
- connect Firebase
- deploy
- publish apps
- use LLMs
- use local models
- self-improve automatically
- choose workflows from memory

These abilities must be added later as separate networks.

## 13. Development Rules

- Keep the engine Python-first.
- Keep the first phases dependency-free.
- Do not add AI before deterministic workflows are solid.
- Do not mix engine files with generated project files.
- Do not create output projects inside the engine by default.
- Prefer `/Volumes/HDD/MM_projects` as the output workspace.
- Every new feature must have a CLI command.
- Every new feature must have verification.
- Every new feature must update memory where appropriate.
- Avoid large Codex prompts.
- Prefer small manual changes.
- Codex may be used only as a limited code executor, not as the architect.
